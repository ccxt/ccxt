CCXT Prediction-Market Unified Types

A proposed set of dedicated structures for the ccxt.prediction namespace. They are not the spot/derivatives Order/Trade/Position — prediction markets have a different shape (an outcome you buy at a probability, not a base/quote pair). Every field below is mapped to what each exchange actually returns in its official API docs / SDK: polymarket, kalshi, limitless, myriad, hyperliquid.

Summary. A three-level hierarchy — Event → Market → Outcome — where the Outcome is the tradeable unit and Order/Trade/Position/Ticker/OrderBook all reference it by its outcome handle. There is no symbol field: the handle is outcome ("MARKET:LABEL") and the raw exchange id is outcomeId. Across all venues, prices are normalized to probability (0–1), amounts to shares, costs to collateral; YES/NO lives inside the outcome handle so side is simply buy/sell. A Market is 1..N outcomes tagged binary | categorical | scalar, so it covers everything from Yes/No to multi-candidate and range markets. Open questions for the team are listed at the end.

How the structures relate

Conventions

  • Prices are probabilities, 0–1. price/bid/ask/last are always in [0, 1] — the implied probability. Each exchange's native units (cents, 1e18 wei, 6-dp micro-units) are normalized in the parser.
  • Amounts are shares; cost is collateral. amount/contracts = number of outcome shares; cost/notional/payout = collateral in the quote currency. A share pays out 1 unit of collateral if its outcome resolves true.
  • Side is buy/sell of an outcome. YES vs NO is encoded in the outcome handle, not in side. (Kalshi's action×outcome and Hyperliquid's A/B collapse to buy/sell.)
  • Identity: outcome is the handle you pass to every method and that keys exchange.outcomes; outcomeId is the raw exchange id used under the hood (it survives a label rename); label is the short human name; market/event are the parents.

Reference structures

Event

The top-level grouping of related markets. Some venues model it first-class (polymarket events, kalshi events+series); others express it as a market group (limitless) or a question (myriad/hyperliquid). When a venue has no grouping, an Event may wrap a single Market.

Market

One question. May be binary (Yes/No), categorical (N>2 outcomes in one market — polymarket/myriad), or scalar (resolves to a number in a range — kalshi). outcomes is therefore 1..N, and marketType tells the consumer which shape to expect. Scalar markets keep the same trading surface as binary; floorStrike/capStrike/strikeType describe the payout curve, and settlementValue carries the realized number once resolved (with the outcome's settleFraction giving the 0–1 payout). executionModel distinguishes a real order book (clob) from pool-priced venues (amm/parimutuel) — see the roadmap note for the latter.

interface PredictionMarket {
    info: any;

Outcome

The tradeable unit — what you actually buy and hold. First-class on polymarket (CLOB tokens[]), myriad, hyperliquid; synthesized as Yes/No on kalshi and limitless. Settlement is general: settleFraction in [0,1] (binary is the 0/1 case).

interface PredictionOutcome {
    info: any;
    outcome: string;        // handle "TRUMP_WIN_2024:YES" — round-trips; ex.outcomes key
    outcomeId?: Str;        // raw exchange/on-chain id (token id / ticker / coin)
    label?: Str;            // short human name "Yes"
    market?: Str; marketId?: Str; event?: Str;

    bid?: Num; ask?: Num;
    active?: Bool;
    winner?: Bool;          // resolved true (the settleFraction === 1 case)
    settleFraction?: Num;   // 0-1 fractional settlement
}

OrderBook

interface PredictionOrderBook {
    outcome: string;     // required — books are per-outcome
    outcomeId?: Str; market?: Str;
    bids: [Num, Num][];  // [probability 0-1, shares]
    asks: [Num, Num][];
    timestamp: Int; datetime: Str;
    nonce?: Int;
}

A real order book is a CLOB concept. polymarket book.bids/asks (per token); kalshi orderbook_fp is dual-sided (yes_dollars + no_dollars), bids-only — each side's asks are the opposite side's bids inverted (ask = 1 − bid); hyperliquid levels[0]/[1]; limitless / myriad CLOB markets. On AMM/parimutuel venues there is no resting book — those expose price via Ticker, and a depth/quote primitive is a later-phase addition (see roadmap). fetchOrderBook stays CLOB-only.

Per-exchange units & normalization

  • polymarket — two APIs: CLOB (snake_case, sizes as 6-dp strings) and Data-API (camelCase, plain numbers); prices already 0–1. Gamma's outcomes/outcomePrices/clobTokenIds are JSON-string arrays (parse them); conditionId joins the two APIs. GET /price documents its side→bid/ask mapping inverted — verify live.
  • kalshi — read the *_dollars (0–1) and *_fp (contracts) fields, falling back to legacy integer cents (÷100). market_type is binary or scalar. Orderbook is dual-sided, bids-only. Multi-candidate questions are an event of binary markets.
  • limitless — USDC and share amounts are scaled by 1e6; orderbook prices are 0–1, but the market-list prices array may be 0–100 — verify. Note realisedPnl (CLOB) vs realizedPnl (AMM).
  • myriad — the order-book path uses 1e18-scaled strings (price = 0–1 fraction ×1e18, amount = shares ×1e18); the AMM / portfolio / market REST responses are human decimals with 0–1 prices. Branch on executionMode / tradingModel.
  • hyperliquid — prediction markets (HIP-4) ride the spot path: side is A/B, a position is a spot balance (no leverage), markPx ≈ probability, and the coin is #<encoding> where encoding = 10·outcome + side. Settlement is fractional (settleFraction).

Decisions & roadmap

Settled

  • Scalar markets stay in the unified Market — same trading surface as binary; marketType:'scalar' + floorStrike/capStrike/strikeType describe the payout curve and settlementValue carries the realized number. Consistent with how CCXT already folds options into Market; no separate type.

Phase 2 — the structure already grows into it, no breaking change

  • AMM / parimutuel. Few venues/methods need it today, so it's deferred. CLOB is the phase-1 focus and fetchOrderBook is CLOB-only (a pool has no resting book). The executionModel discriminator (clob/amm/parimutuel) is already on Market, so phase 2 can add a fetchQuote(outcome, side, amount|cost) method returning a small PredictionQuote (avg price + price impact) without changing any existing structure.

Open

  • Order-by-cost. Prediction users think in collateral ("bet $10"), not share count — createOrder(outcome,'buy',10,0.62) spends $6.20, not $10. Add a createOrderByCost / params.cost path, or at least document the unit loudly?
